# v0.10.0 Release Plan

## Purpose

`v0.10.0` should be the first clearly consolidated Makrell family release.

It should not try to mean:

- feature-complete
- stable `1.0`
- final language design

It **should** mean:

- the family has a coherent current shape
- the main implementations are installable and testable
- editor support is being treated as part of the product, not an afterthought
- documentation, specs, packaging, and tooling are aligned enough to give a
  good developer experience with existing functionality
- the family has a shared async/await surface for the main implementation
  tracks

## Release theme

**Consolidation, packaging, and developer experience.**

This release is about making the current Makrell family usable and legible.

## Release scope

The `v0.10.0` milestone should cover:

- packaging and install paths for Python, TypeScript, `.NET`, and the VS Code extension
- a current feature/status matrix
- better shared conformance/fixture coverage for family formats
- a stronger `vscode-makrell` experience
- documentation and site updates that reflect what is actually real today
- shared async/await support in MakrellPy, MakrellTS, and Makrell#

It should avoid large new design tracks unless they are directly needed for
consolidation.

Async/await is the one clear language-surface exception for this release. It is
important enough to developer experience and host interop that it should be
treated as in-scope for `v0.10.0`.

For this release, compile-time macro execution should also be treated as part
of consolidation rather than an optional advanced feature. The current family
still has visible differences between what runtime Makrell can do and what
compile-time Makrell can do, especially outside MakrellPy. `v0.10.0` should
close that gap by moving toward the same parser and compiler path for meta and
runtime code, not by growing a second long-lived compile-time sublanguage.
For Makrell# in particular, this should be treated as a hard architectural
direction: meta must not become a separate implementation of Makrell#.

## What counts as in-scope

### 1. Packaging and release readiness

#### MakrellPy

- audit `setup.py` / `pyproject.toml`
- make metadata current
- verify install and wheel build
- verify CLI entry points
- write a minimal release procedure

#### MakrellTS

- make `impl/ts/package.json` publish-ready
- decide package exports and package files
- verify `bun run build`
- define package/install usage in docs

#### Makrell#

- decide which `.csproj` files are packageable
- add NuGet metadata where needed
- decide whether `MakrellSharp.Cli` becomes a packaged `.NET` tool for this release
- verify `dotnet pack` or explicitly defer it

#### VS Code extension

- refresh package metadata
- make repo/homepage URLs current
- verify packaging command
- define marketplace/release procedure
- extract reusable language/editor assets so future web tooling can share them
  instead of duplicating them
- move toward a TS-based family language-server/tooling path that can support
  Makrell, MakrellPy, MakrellTS, Makrell#, MRON, MRML, and MRTD from one
  editor-oriented architecture

### 2. Family status clarity

- create one release-level feature matrix
- include:
  - MBF
  - Makrell language support
  - MRON
  - MRML
  - MRTD
  - packaging
  - editor support
  - CLI/tooling
- clearly distinguish:
  - implemented
  - partial
  - evolving
  - planned

### 3. Conformance and fixtures

- strengthen shared or mirrored fixtures for:
  - MRON
  - MRML
  - MRTD
- make drift more visible across Py/TS/.NET
- prefer fixture-based parity over ad hoc examples

### 4. VS Code extension refresh

This is a release-blocking workstream for `v0.10.0`.

The extension should support the family more honestly and more usefully.

Minimum target:

- file support for:
  - `.mr`
  - `.mrpy`
  - `.mrts`
  - `.mrsh`
  - `.mron`
  - `.mrml`
  - `.mrtd`
- refreshed syntax highlighting
- family-aware snippets
- command palette support for current workflows where practical
- basic run/build/check commands for the three main language tracks where
  practical:
  - MakrellPy
  - MakrellTS
  - Makrell#
- a clearer TS-first architecture for future work
- an architecture that allows future sharing of grammar/snippets/language
  metadata with a web playground rather than duplicating them
- no hard dependency on the Python VS Code extension
- optional `makrell-langserver` integration for interim hover/go-to/diagnostics
  while the longer-term TS family tooling path is being built
- a clear direction that deeper editor support should converge on one
  TypeScript-based family language-server path, including the data formats
  rather than treating them as second-class files
- an interim path for Makrell# compiler diagnostics so the `.NET` track has a
  basic editor-intelligence story before the TS-family language server is ready
- editor-visible diagnostics with code markings for all six current family file
  types:
  - MakrellPy / `.mrpy`
  - MakrellTS / `.mrts`
  - Makrell# / `.mrsh`
  - MRON / `.mron`
  - MRML / `.mrml`
  - MRTD / `.mrtd`

### 5. Documentation and site

- root README should match current family shape
- implementation READMEs should show current install and workflow paths
- `makrell.dev` should reflect the real family structure and current tooling
- specs should describe implemented behaviour accurately enough for the release
- `v0.10.0` should explicitly surface a small set of showcase examples for all
  three main implementation tracks, not leave the most interesting material
  buried in tests or scattered examples

### 6. Compile-time capability parity

This is also a release-level concern for `v0.10.0`.

Compile-time Makrell should not feel like a separate toy language in some
tracks and the real language in others.

Minimum target:

- MakrellPy, MakrellTS, and Makrell# should each have a credible compile-time
  story for the shared `pipe` / `rpn` / `lisp` showcase set
- compile-time code should use the same parser and, as far as practical, the
  same compiler/runtime path as ordinary code
- pattern matching and other important existing language features should be
  available in compile-time code, or clearly documented as a release gap
- the remaining meta/runtime differences should be explicit in docs rather than
  discovered accidentally by users
- Makrell# meta should not be treated as a second language implementation; any
  remaining meta-specific code should be orchestration/bootstrap infrastructure,
  not a separate semantic engine

### 7. Shared async/await support

This is also an explicit release requirement for `v0.10.0`.

Minimum target:

- MakrellPy, MakrellTS, and Makrell# all support:
  - `{async fun ...}`
  - `{await expr}`
- the syntax follows Makrell conventions rather than copying raw host syntax
- host-specific extensions such as `{async for ...}` or `{async with ...}` are
  documented clearly if they differ by track
- examples and tests exist in the main tracks

## What is intentionally out of scope

These are good ideas, but should not be required for `v0.10.0`:

- `mx-path`
- major new macro systems
- new large miniformats beyond consolidating MRTD
- large language-surface expansions
- full cross-host parity in all advanced semantics
- full formatter or full language server redesign
- a full web playground implementation
- advanced async design beyond the shared core forms

These can continue as post-`0.10.0` work.

## Discussion items

These are important enough to keep visible during the `v0.10.0` cycle, but are
not yet committed release requirements.

- Whether Makrell# should surface a stronger typing story in `v0.10.0`, either
  through language forms, emitted-code ergonomics, interop guidance, or a
  clearer public position on typed vs dynamic Makrell# usage.
- Whether `v0.10.0` should explicitly position one TS-based family language
  server as the long-term editor/runtime-neutral tooling direction for
  Makrell, MakrellPy, MakrellTS, Makrell#, MRON, MRML, and MRTD.

## Suggested execution order

### Phase 1: packaging baseline

Do first:

- Python packaging cleanup
- TS packaging cleanup
- `.NET` package/tool decision
- VS Code extension packaging audit

Reason:

This gives a concrete definition of what `0.10.0` can actually ship.

### Phase 2: VS Code refresh baseline

Do next:

- file associations
- grammar refresh
- snippets
- README/package cleanup

Reason:

This is one of the most visible gaps in the current developer experience.

### Phase 3: conformance and status matrix

Do next:

- MRON/MRML/MRTD fixtures
- release-level feature matrix
- compile-time/runtime parity review across Py/TS/.NET

Reason:

This makes the family shape explicit and helps prevent accidental drift.

### Phase 4: docs/site alignment

Do next:

- root README
- implementation READMEs
- `makrell.dev`
- release notes draft

Reason:

At this point the docs can describe a relatively stable release scope.

### Phase 5: release verification

Final pass:

- packaging smoke tests
- extension packaging smoke
- repo-wide core test runs
- final docs check

## Acceptance criteria

`v0.10.0` is ready when all of these are true:

- the main implementation tracks have a credible install/package story
- `vscode-makrell` reflects the actual family much better than before
- the family status picture is explicit in docs
- compile-time Makrell is no longer obviously underpowered in TS/`.NET`
- shared format behaviour is backed by meaningful fixtures/tests
- the repo and site give a coherent first-run experience

## Risks

### 1. Too much scope

There is a risk of quietly turning `0.10.0` into a broad platform rewrite.

Mitigation:

- keep the milestone focused on consolidation
- defer large new language features

### 2. Packaging decisions may expose unresolved boundaries

Especially in `.NET` and TS, package boundaries may still be fuzzy.

Mitigation:

- write down explicit package/tool decisions even if some are deferred
- prefer clarity over fake completeness

### 3. VS Code work can expand quickly

Editor tooling can easily become its own large project.

Mitigation:

- define a minimum useful extension baseline for `0.10.0`
- leave full LSP/formatter ambitions for later

### 5. Async may expand into a large semantic track

Mitigation:

- keep `v0.10.0` focused on the shared core forms
- write down host/profile distinctions clearly
- avoid overdesigning advanced async constructs before the common syntax lands

### 4. Playground work can sprawl into a second tooling stack

Mitigation:

- record the playground as a likely `0.10.0` companion track, but do not force
  full implementation before the design profile exists
- make `vscode-makrell` shareable rather than cloning its editor/language data

## Suggested release note framing

Possible framing:

> Makrell `v0.10.0` is the first release focused on consolidating the family:
> packaging, tooling, editor support, format coverage, and clearer cross-runtime
> documentation.

That is a better story for this milestone than claiming a wave of new language
features.

## Implementation showcases for `v0.10.0`

`v0.10.0` should not only say that the Makrell family is coherent; it should
show a few concrete examples that make each implementation track memorable.
These should be existing strengths, not new features invented just for the
release.

### MakrellPy

MakrellPy already has a few macro examples that are unusually good at showing
what the language can do without needing a large framework around them.

Source today:

- `impl/py/tests/makrellpy/test_meta.mr`
  - `pipe`
  - `rpn`
  - `lisp`

Recommended framing:

- `pipe`
  - ergonomic syntax reshaping
  - takes an ordinary computation and rewrites it into readable pipeline form
- `rpn`
  - alternative notation through macros
  - demonstrates direct AST construction from postfix input
- `lisp`
  - language embedding and translation
  - demonstrates that a different surface notation can be hosted inside Makrell

For `v0.10.0`, these should not be Python-only curiosities. They should become
a shared macro showcase set for the family wherever feasible:

- `pipe`
- `rpn`
- `lisp`

That gives the release one recognisable cross-implementation story for macros
instead of three unrelated examples.

### MakrellTS

MakrellTS should be showcased as the reference implementation and the strongest
cross-runtime story in the family.

Good current showcase areas:

- browser runtime and isolated meta execution
- typed output surface such as `compileToTs` / `compileToDts`
- one compact macro example that also runs in the TS track

Concrete candidate examples today:

- `impl/ts/examples/hello.mrts`
  - compact macro + pattern-matching demo
  - good as a short README/release-notes example
- `impl/ts/examples/nbody-browser/app.mrts`
  - good as the "Makrell in the browser" example
  - better as a linked showcase than as an inline snippet
- `impl/ts/examples/browser-compile/`
  - good for demonstrating in-browser compile/execute flow

Recommended framing:

- "Makrell that compiles, runs, and typechecks in the TS ecosystem"
- "Same family syntax, but with a browser and TS-native tooling story"

For `v0.10.0`, MakrellTS should ideally also gain public `pipe`, `rpn`, and
`lisp` showcase examples so the shared macro trio exists here as well, not only
in MakrellPy.

### Makrell#

Makrell# should be showcased for things that feel distinctly `.NET` rather than
for trying to imitate the Python track exactly.

Good current showcase areas:

- CLR interop with Makrell-shaped generic type syntax
- pattern matching
- compile/load/build flow through the CLI

Concrete candidate examples today:

- `impl/dotnet/examples/interop.mrsh`
  - compact CLR interop example
  - good for a short public snippet
- `impl/dotnet/examples/macros.mrsh`
  - small macro/meta example
  - useful as a secondary example, not the headline one
- `impl/dotnet/README.md` pattern-matching examples
  - currently stronger than the checked-in example files for showing the
    distinctiveness of Makrell#
  - should probably be turned into one dedicated public example file for the
    release
- CLI flow already exercised by:
  - `build`
  - `run-assembly`
  - `emit-csharp`
  - `meta-sources`

Recommended framing:

- "Makrell as a compact front-end to the CLR"
- "Makrell syntax with real .NET interop and assembly generation"

For `v0.10.0`, Makrell# should ideally also gain public `pipe`, `rpn`, and
`lisp` showcase examples so the shared macro trio exists here as well, not only
in MakrellPy.

This should not be achieved only through one-off showcase hacks. The release
should also move Makrell# toward the normal parser/compiler path for
compile-time execution so the examples reflect where the implementation is
heading.

### Recommended placement

- release notes section:
  - "One shared macro showcase, plus three implementation-specific examples"
- `makrell.dev`
  - one short showcase block per implementation
- implementation READMEs or dedicated examples pages
- optional homepage callout linking into a family showcase page

### Recommended packaging of examples

- move or copy public showcase examples into dedicated example/showcase
  locations rather than citing tests directly
- keep tests as regression coverage
- present the public versions with a short explanation of what each example is
  showing
- keep each example small enough to be readable in release notes or on the site

Suggested minimal deliverables:

- MakrellPy:
  - one public showcase file or page containing `pipe`, `rpn`, and `lisp`
- MakrellTS:
  - one public showcase file or page containing `pipe`, `rpn`, and `lisp`
  - one small syntax/macro showcase from `hello.mrts`
  - one linked browser showcase from `nbody-browser` or `browser-compile`
- Makrell#:
  - one public showcase file or page containing `pipe`, `rpn`, and `lisp`
  - one compact interop example
  - one compact pattern-matching example

Preferred release shape:

- shared across all three:
  - `pipe`
  - `rpn`
  - `lisp`
- implementation-specific:
  - MakrellPy:
    - macro-friendly language-embedding examples beyond the shared trio
  - MakrellTS:
    - browser/runtime and TS-ecosystem examples
  - Makrell#:
    - CLR interop, build/load flow, and pattern matching

This is useful for `v0.10.0` because it highlights real existing capability
across the family rather than adding new language surface just for promotion.

## Likely companion project: MakrellTS playground

A web-based MakrellTS playground with examples, docs, REPL, and an editor is a
strong fit for the broader `v0.10.0` story, but it should be treated carefully.

Current direction:

- MakrellTS-first
- browser-native
- examples/docs/REPL/editor in one surface
- reuse code or data from `vscode-makrell` where practical
- avoid duplicating grammar, snippets, or language metadata

This means the extension refresh should already lean toward shared TS modules
for:

- language metadata
- snippets data
- grammar assets where shareable
- example manifests

The playground itself can be planned in more detail once a design profile
exists.

## First concrete tasks

If work should start immediately, do these first:

1. update `vscode-makrell/package.json` scope and language support plan
2. audit `impl/ts/package.json` for npm readiness
3. audit `impl/py` metadata/release path
4. decide `.NET` package and CLI publication scope for `0.10.0`
5. write the feature/status matrix

Those five tasks will define the rest of the milestone much more clearly.
